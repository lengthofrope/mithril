## ADR-003: Prefix all API route names with `api.`

**Date:** 2026-03-07
**Phase:** 4
**Tags:** backend, routing, api
**Status:** Accepted

### Context

The application registers both web routes (`routes/web.php`) and API routes (`routes/api.php`). Laravel's `apiResource()` in `api.php` generated route names like `tasks.index`, `tasks.store`, `bilas.index`, etc. — identical to the names used by web routes in `web.php`.

Because API routes are loaded after web routes, the API route names silently overwrote the web route names. This caused:

1. `route('tasks.index')` in Blade templates resolved to `/api/v1/tasks` instead of `/tasks`, breaking the filter-manager AJAX requests (they hit the API endpoint, got a JSON response or redirect instead of an HTML partial).
2. `route('tasks.store')` resolved to the API store endpoint, so the dashboard quick-add form and the tasks page inline form both failed (POST to API returned JSON, not a redirect).
3. Several route names generated by `apiResource()` (e.g. `tasks.show`, `notes.show`, `notes.store`, `notes.update`) had no corresponding web route. Views referenced these names and they happened to work only because the API resource provided them — but they linked to API endpoints, not web pages.

### Decision

1. **Add `->as('api.')` to the API route group** in `routes/api.php`. All API resource routes are now prefixed: `api.tasks.index`, `api.tasks.store`, `api.bilas.index`, etc. This eliminates the name collision with web routes.

2. **Add missing web routes** that views actually reference:
   - `tasks.show` → `TaskPageController@show`
   - `notes.show` → `NotePageController@show`
   - `notes.store` → `NotePageController@store`
   - `notes.update` → `NotePageController@update`

3. **Web controllers own all Blade-facing routes.** Any route name referenced in a Blade template must be defined in `web.php` and handled by a Web controller. API routes are exclusively for JavaScript/fetch consumers and must always be referenced with the `api.` prefix.

### Consequences

- **API route references in TypeScript/JS** that use named routes (e.g. via Ziggy or hardcoded names) must be updated to include the `api.` prefix. Currently the frontend uses hardcoded URL paths (e.g. `/api/v1/tasks`), so no JS changes were needed.
- **New web controller methods** were added to `NotePageController` (`show`, `store`, `update`) and `TaskPageController` (`show`). New Blade views were created: `pages/tasks/show.blade.php` and `pages/notes/show.blade.php`.
- **Test impact:** All 598 existing tests pass. The route name collision had previously caused test failures for any test that rendered views referencing the affected route names.

### Follow-ups / open questions

- The `tasks.show` and `notes.show` pages are minimal placeholders. They should be fleshed out with auto-save, inline editing, and related data once those features are prioritised.
- Consider adding a CI check or test that asserts no route name in `web.php` is also defined in `api.php`, to prevent future collisions.
